# MediaPipe 메이크업+헤어 오버레이 POC — 검증 보고서

> 결론 먼저: **얼굴 100% 보존 = 검증됨.** 립·블러셔 메이크업과 헤어 컬러
> 변경 모두 브라우저 온디바이스로 실제 동작. Gemini의 "다른 사람 얼굴로
> 바뀜" 문제가 **구조적으로 불가능**한 방식임을 실제 결과물로 확인.
> 단, 운영자 핵심 우려대로 **모자를 쓰면 헤어 컬러는 보이는 머리카락에만**
> 적용된다(메이크업은 영향 없음).

---

## 1. 무엇을, 어떻게 검증했나

- **환경**: React+Vike 본체는 **전혀 안 건드림**. 별도 브랜치
  `poc/mediapipe-overlay` + 폴더 `experiments/mediapipe-poc/`에 독립 구현.
  Gemini 코드 그대로 → 즉시 롤백 가능.
- **기술 스택**: `@mediapipe/tasks-vision@0.10.18` (CDN 로드, 번들 0 증가)
  - `FaceLandmarker` (face_landmarker.task) → 478 랜드마크 → 입술/볼 영역
  - `ImageSegmenter` (selfie_multiclass_256x256) → 픽셀 분류
    (0 배경 / **1 머리카락** / 2 몸-피부 / 3 얼굴-피부 / 4 옷 / 5 기타)
- **검증 방식(중요)**: 말로만이 아니라 **실제로 돌렸다.** Playwright로
  헤드리스 Chromium을 띄워 실제 사진을 파이프라인에 통과시키고, 나온
  PNG를 직접 눈으로 확인. 모델은 CPU delegate(XNNPACK)로 구동.
- **테스트 이미지 3장**:
  1. `frontal_1` — 정면, 적갈색 머리 (여성)
  2. `frontal_2` — 약 15~20° 돌아간 얼굴, 갈색 머리
  3. `hat_beanie` — **분홍 비니로 머리 대부분을 덮은** 3/4 각도 (= 운영자 셀카 상황)

## 2. 핵심 결과 (실측치)

| 케이스 | 얼굴 검출 | 머리카락 픽셀% | 처리시간(CPU) |
|---|---|---|---|
| frontal_1 (정면) | ✅ | 37.0% | 1.5s |
| frontal_2 (각도) | ✅ | 38.9% | 1.0s |
| hat_beanie (모자) | ✅ | **11.1%** | 0.8s |

- 모델 최초 로딩: **4.6초** (이후 브라우저 캐시 → 재방문 거의 즉시)
- 처리: 이미지당 **0.8~1.6초** (헤드리스 CPU 기준. 실제 폰 GPU면 더 빠름)

### 결과 이미지 (`out/` 폴더)
- `*_result.png` — 오버레이 적용 결과
- `*_diff.png` — **원본 대비 바뀐 픽셀만 빨강** (= 보존 증거)

## 3. 검증 포인트별 판정

**A. 얼굴 100% 보존 → ✅ 완전 통과 (이게 핵심)**
- 방식 자체가 "원본 픽셀을 그대로 깔고, 입술/볼/머리 영역에만 합성".
  얼굴을 **재생성하지 않는다.** Gemini와 근본적으로 다름.
- `*_diff.png`에서 변경 픽셀이 **머리카락 + 입술 + 볼 두 곳에만** 몰려 있고
  눈·코·입 모양·이마·피부·배경은 전부 그대로(검정). → 보존이 코드가 아니라
  **이미지로 증명**됨.

**B. 메이크업(립+블러셔) → ✅ 통과**
- 립: 입술 외곽 링 - 내곽 링(evenodd)으로 입술 밴드에만 multiply 틴트 + 페더링.
- 블러셔: 볼 중심 라디얼 그라데이션. 1차엔 "동그란 자국"이 보였으나
  알파 0.42→0.22, 반경 확대, blur 6px 페더로 **자연스러운 발색**으로 개선.
- 약 15~20° 돌아간 얼굴(frontal_2, 비니 여성)에서도 볼/입술 위치 정확.

**C. 헤어 컬러 변경 → ✅ 통과 (단, 조건부)**
- 휘도(밝기) 보존 리컬러 → 머리카락 **결·하이라이트가 그대로 살아있는** 자연스러운
  염색 느낌. frontal_2를 보라색으로 바꿔도 가닥 질감이 유지됨.
- 세그멘테이션이 **비니(니트 모자)를 머리카락으로 오인하지 않음** → 모자는
  색이 안 바뀌고, 모자 밑으로 보이는 머리카락만 코퍼로 변경. (정확한 분리)

**D. 모자/측면 → ⚠️ 한계 확인 (= 운영자 우려 그대로)**
- **모자**: 비니 케이스 머리카락 픽셀이 **11%**(맨머리 37% 대비 1/3 미만).
  즉 머리가 모자에 가려지면 **염색할 머리 자체가 없다.** 보이는 가닥만 바뀜.
  → **메이크업은 모자와 무관하게 잘 됨.** 헤어 컬러만 제약.
- **각도**: 3/4 각도까지는 랜드마크/세그 모두 정상. 완전 90° 측면은 이번
  공개 테스트셋으로 구하지 못해 미검증 → **운영자 실제 측면 셀카로 1회 확인 권장**
  (FaceLandmarker는 극단 측면에서 정확도가 떨어지는 알려진 한계 있음).

## 4. 남은 한계점 (전체 구현 시 다듬을 것)

1. **모자 헤어**: 근본 한계. 대응책 — 모자 감지 시 "헤어 컬러는 보이는 부분만
   적용됩니다" 안내, 또는 모자 케이스는 헤어 옵션 자동 숨김.
2. **다인 사진**: 메이크업은 가장 큰 얼굴 1명만(numFaces=1). 그러나 세그멘테이션은
   **화면 속 모든 사람 머리**를 칠함 → 셀카(1인)면 무관하나, 다인 사진은 검출된
   얼굴 주변으로 헤어 마스크를 제한하는 후처리 필요.
3. **헤어/옷 경계 오검출**: 퍼 카라 등에서 약간 머리카락으로 새는 픽셀 발견 →
   confidence threshold + 마스크 erosion으로 정리 가능.
4. **립 디테일**: 입을 크게 벌린 사진에선 치아에 살짝 틴트가 묻을 수 있음(현재
   evenodd로 대부분 방지). 내곽 마스크 정밀화로 해결.
5. **CDN 의존**: 모델/wasm을 구글·jsdelivr에서 로드. 오프라인/차단 환경 대비
   `public/`에 self-host 옵션 검토(약 8~12MB).

## 5. 전체 구현 난이도 / 기간 예상

핵심 기술 리스크("AR 오버레이로 얼굴 보존 + 머리 염색이 진짜 되나?")는
**이번 POC로 해소됨.** 남은 건 엔지니어링 + 다듬기.

| 작업 | 난이도 | 예상 |
|---|---|---|
| 여 6 / 남 6 색조·파라미터 테이블 (색·알파만) | 낮음 | 0.5일 |
| 1-of-6 선택 UI + AnalysisApp 통합 | 중 | 1~1.5일 |
| 블러셔/립/헤어 경계 리얼리즘 폴리시 | 중(=주 작업) | 1~2일 |
| 모자·다인·측면 엣지케이스 처리 + 안내 UX | 중 | 1일 |
| Gemini 경로 제거 + QA | 낮음 | 0.5일 |

**합계 약 4~5일** (POC 엔진이 이미 있으므로). 서버 비용 **$0**(온디바이스),
Gemini 이미지 생성 API 비용 **제거**가 부가 이득.

## 6. 운영자가 직접 확인하는 법

```bash
cd experiments/mediapipe-poc
# (A) 헤드리스 재현 — 이미 설치돼 있음
node run-poc.mjs        # out/ 에 결과·diff PNG 생성

# (B) 브라우저에서 본인 셀카로 직접 (권장: 모자/측면 셀카 테스트)
python3 -m http.server 8000
#  → http://localhost:8000  접속 → 사진 업로드 → 색 골라 [처리]
#  → "헤어 마스크 보기"로 머리 인식 영역 확인 가능
```
**특히 운영자 본인의 모자/측면 셀카를 (B)로 넣어보는 것이 마지막 검증.**

## 7. 권고

- 메이크업(립·블러셔 등)은 **지금 방식으로 전면 채택 추천** — 얼굴 보존 완벽.
- 헤어 컬러는 **"맨머리 셀카에선 훌륭, 모자면 제한적"**임을 UX로 솔직히 안내하면
  충분히 상품성 있음.
- 2단계(전체 구현) 진행 시, 위 4~5일 일정 + 폴리시 우선순위로 착수 가능.
